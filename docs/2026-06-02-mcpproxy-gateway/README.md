# Dockerized MCPProxy Gateway

## Purpose

MCPProxy is the shared gateway for standards-compliant stdio, HTTP, SSE, and Streamable HTTP MCP
servers. MCP-capable hosts connect to one downstream endpoint while MCPProxy manages upstream OAuth
credentials separately. Models do not connect to MCP directly; their host application does.

Fastmail is the initial upstream. Client wiring is deliberately deferred until the gateway passes
its local and private-HTTPS smoke tests.

## Network and authentication

The container uses host networking because MCPProxy creates an ephemeral OAuth callback listener on
`127.0.0.1`. Run initial OAuth login from the devserver host and complete authorization in a browser
on that same host.

The normal HTTP API listens only on `172.19.0.1:3130`, the host gateway address for `mybridge`.
Caddy can reach that address from its container. Port `3130` is not bound on the LAN interface.

Authentication has two separate boundaries:

- `MCPPROXY_API_KEY` is the administrative secret rendered from
  `op://clankers/mcpproxy-admin/password`.
- MCP clients use a scoped agent token generated by MCPProxy and stored at
  `op://clankers/mcpproxy-agents/password`.
- Fastmail receives only MCPProxy's OAuth access token. Downstream bearer tokens are never passed
  through to Fastmail.

## Deploy

Build the `projects/dockers` image branch first. Once it is merged into the canonical checkout:

```bash
just rs
just mcpproxy-up
just mcpproxy-health
just mcpproxy-auth-fastmail
just mcpproxy-auth-status
just mcpproxy-token-create
```

`just mcpproxy-auth-fastmail` prints the Fastmail authorization URL emitted by the headless daemon.
Open it in a browser running on the devserver host so the loopback callback can complete.

The token-create command prints the raw downstream token once. Store it as the password field of
the `clankers/mcpproxy-agents` 1Password item, then run:

```bash
MCPPROXY_AGENT_TOKEN="$(op read 'op://clankers/mcpproxy-agents/password')" just mcpproxy-smoke
```

## Endpoints

| Endpoint | Purpose |
|---|---|
| `https://mcp.dev.ankitson.com/mcp` | Default direct route for ordinary MCP clients |
| `https://mcp.dev.ankitson.com/mcp/all` | Explicit direct route |
| `https://mcp.dev.ankitson.com/mcp/call` | Retrieval route for large catalogs and model tool limits |

The private Caddy route intentionally does not expose `/ui/`, `/api/v1/*`, or `/mcp/code`.

## Persistent state

`config/mcpproxy.seed.json` is copied only when `/data/mcp_config.json` does not exist. After first
boot, treat the `devserver_mcpproxy_data` volume as the live source of truth. It contains
configuration, DCR credentials, OAuth refresh tokens, and hashed downstream agent tokens.

Back up that volume only into encrypted storage. Restoring the volume restores Fastmail OAuth and
downstream token state together.

## Verification

1. Confirm an unauthenticated MCP request returns `401`.
2. Complete Fastmail OAuth and verify `just mcpproxy-auth-status`.
3. Generate and store the scoped agent token.
4. Run authenticated `initialize` and `tools/list` requests through local and private HTTPS routes.
5. Call a harmless Fastmail read tool, restart `mcpproxy`, and repeat the read call.
6. Confirm `/ui/` and `/api/v1/status` return `404` through Caddy and port `3130` is absent from the
   LAN interface.

The previous LiteLLM Fastmail experiment stays isolated on its existing POC branch. Remove the
orphaned `fastmail-mcp` bridge container only after this gateway passes smoke tests.
